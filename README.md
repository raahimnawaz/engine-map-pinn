# Engine-Map PINN — physics-informed surrogate of a V12 engine map

[![CI](https://github.com/raahimnawaz/engine-map-pinn/actions/workflows/ci.yml/badge.svg)](https://github.com/raahimnawaz/engine-map-pinn/actions/workflows/ci.yml)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)
[![Python 3.11+](https://img.shields.io/badge/python-3.11%2B-blue.svg)](https://www.python.org/)

A **physics-informed neural network** that reconstructs an engine's full
torque / power / fuel map from only a handful of sparse dyno sweeps — and beats
a data-only network by **~65%**, almost entirely by getting the *unsampled*
regions right.

The engine modeled is the **Lamborghini Aventador SVJ** — a 6.5 L naturally
aspirated V12. The first-principles plant is calibrated to the car's real
published numbers, then a PINN learns a surrogate of it from noisy, sparse
samples the way a real dyno session would produce them.

> **Scope / honesty:** the "data" is generated by a mean-value physics model
> calibrated to published SVJ peak figures — there is no proprietary dyno data
> here, and none exists publicly for a specific car. A **twin-turbo "build"**
> variant is included as a physically-plausible *what-if* (boost layered on the
> calibrated NA block), **not** a measurement of any real tuned car.

## Headline result

![PINN vs data-only baseline](figures/pinn_vs_baseline.png)

The dyno data covers only **mid-to-high throttle (50–100%)**; the low-load band
is left unsampled on purpose. A data-only network has no idea what happens
there and its error explodes; the PINN's physics residuals reconstruct it.

| Torque MAE vs ground truth | Data-only baseline | PINN (data + physics) |
|---|---:|---:|
| Whole map | 13.2 N·m | **4.8 N·m** (−64%) |
| Unsampled low-load region | 23.3 N·m | **6.3 N·m** (−73%) |

## BSFC island + optimal operating line

![BSFC island](figures/bsfc_island.png)

The brake-specific-fuel-consumption "island" and the **optimal operating
line** — for any power demand, the most-efficient (rpm, throttle) point. This
is exactly the curve a hybrid-powertrain energy manager or a CVT shift schedule
is built around, which is what makes the map more than a pretty picture.

## How the PINN works

The network maps `(rpm, throttle) → (brake torque, fuel rate)`. Brake power is
the **exact** identity `P = τ·ω`, so it is *derived*, never a free output — the
architecture itself bakes in that physics. On top of the sparse data, a residual
loss enforces the engine's governing structure on dense collocation points:

1. **Affine-in-load** — `∂²τ/∂throttle² ≈ 0`. A mean-value engine's brake torque
   is affine in throttle at fixed rpm (torque ∝ trapped air charge ∝ MAP). This
   is the value-pinning prior: it lets the high-throttle sweeps *determine* the
   unsampled low-load region by extrapolating the line, not guessing.
2. **Load monotonicity** — `∂τ/∂throttle ≥ 0`.
3. **Fuel affine in load** — `∂²(fuel)/∂throttle² ≈ 0` (Willans-line structure).
4. **Thermodynamic bound** — brake power ≤ fuel power (efficiency < 1).

The data-only baseline is the *same network* with the residual term switched
off, so the comparison isolates exactly what the physics buys you.

## Reproduce

```bash
uv venv --python 3.12 && uv pip install -e ".[dev]"
PYTHONPATH=src python scripts/make_figures.py   # regenerates every figure + prints metrics
uv run pytest -q                                # physics-calibration + PINN-beats-baseline tests
```

## Layout

```
src/enginemap/
  engine.py     first-principles mean-value V12 (NA + twin-turbo), auto-calibrated to specs
  dataset.py    sparse noisy "dyno" sampler + dense ground-truth grid
  pinn.py       PINN (torque+fuel heads, P=τω built in) + autograd physics residuals + baseline
scripts/
  make_figures.py   end-to-end: calibrate → sample → train PINN vs baseline → all figures
tests/          engine hits published peaks; PINN beats the data-only net
figures/        generated results
```
